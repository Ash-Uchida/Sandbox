## 1. Problem Statement

LexCursor handles privileged legal documents, but anyone who reaches the app can
see every screen — there is no concept of a user, login, or session. Without
authentication we cannot protect data, attribute actions, or satisfy SOC 2 /
client confidentiality obligations, which blocks every data-bearing feature.
This story adds login, session management, and route protection.

## 2. Goals

- Let users sign up, log in, and log out securely.
- Protect all app routes so only authenticated users can access them.
- Persist a real `User` record and expose the current user to the app + API.
- Establish the authn/authz seam that RBAC (F1.4) and audit logging (P5.3) build on.

## 3. Non-Goals

- Organization/workspace modeling and role-based permissions (F1.4).
- SSO/SAML, SCIM provisioning, MFA hardware keys (future stories).
- Billing, plans, or seat management.

## 4. Personas & User Stories

- **As an associate attorney**, I want to log in so that only I (and authorized
  colleagues) can open my contracts.
- **As a partner / reviewer**, I want assurance that access requires
  authentication so client confidentiality is preserved.
- **As a legal-ops admin**, I want users to have real accounts so activity can be
  attributed (groundwork for audit logging).
- **As a developer**, I want `getCurrentUser()` helpers on server and client so
  features can gate on identity easily.

## 5. Functional Requirements

- **FR-1.** The system MUST provide email/password signup and login, and SHOULD
  support at least one OAuth provider (Google).
- **FR-2.** The system MUST create/persist a `User` row on first successful auth.
- **FR-3.** All routes except the public auth pages MUST require an authenticated
  session; unauthenticated requests MUST redirect to `/login`.
- **FR-4.** The system MUST provide logout that fully invalidates the session.
- **FR-5.** Sessions MUST be httpOnly, Secure, SameSite cookies with a defined
  expiry and refresh/rotation policy.
- **FR-6.** Passwords (if stored) MUST be hashed with a modern KDF (bcrypt/argon2);
  plaintext MUST never be logged or stored.
- **FR-7.** The API MUST expose the current user to server components and route
  handlers via a single helper.
- **FR-8.** The top bar MUST show the real signed-in user and a logout control.

## 6. Non-Functional Requirements

- **Performance** — auth check (session → user) p95 < 50ms server-side; login
  round-trip p95 < 1s.
- **Security** — OWASP ASVS-aligned: CSRF protection, rate-limited login, generic
  error messages (no user enumeration), secure cookie flags, secret rotation.
- **Privacy & Compliance** — SOC 2 (access control), GDPR/CCPA (lawful basis,
  deletion path), attorney–client privilege (no privileged data in auth logs).
- **Accessibility** — auth forms WCAG 2.1 AA: labeled fields, error association,
  keyboard-navigable, visible focus.
- **Scalability** — stateless verification where possible; session store scales
  horizontally.
- **Reliability** — auth outage fails closed (deny access), never fails open.
- **Observability** — emit auth events (login success/failure, logout) with
  user id + IP, excluding secrets.
- **Maintainability** — single auth module; no scattered session logic.
- **Internationalization** — auth copy externalized.
- **Backward compatibility** — N/A — first auth implementation.

## 7. Acceptance Criteria

- **AC-1.** *Given* a logged-out user, *When* they visit `/dashboard`, *Then* they
  are redirected to `/login`.
- **AC-2.** *Given* valid credentials, *When* the user logs in, *Then* a session
  cookie is set and they land on `/dashboard` as the signed-in user.
- **AC-3.** *Given* a logged-in user, *When* they click logout, *Then* the session
  is invalidated and `/dashboard` again redirects to `/login`.
- **AC-4.** *Given* repeated failed logins, *When* a threshold is exceeded, *Then*
  further attempts are rate-limited.
- **AC-5.** *Given* an authenticated session, *When* a server component calls the
  current-user helper, *Then* it returns the correct `User`.

## 8. Data Model

- New table `users` — `id` (uuid pk), `email` (unique, citext), `name`,
  `image_url`, `password_hash` (nullable for OAuth-only), `created_at`,
  `updated_at`, `last_login_at`.
- Session storage per chosen provider (provider-managed table or cookie/JWT).
- Indexes: unique index on `email`.
- Migration: `prisma/migrations/<timestamp>_add_users/migration.sql`.
- Backfill: none (new table).

## 9. API Surface

- Auth routes provided by the chosen library (e.g. `/api/auth/*` for Auth.js).
- `GET /api/me → { id, email, name, imageUrl }` for the current user (401 if
  unauthenticated).
- Server helper: `getCurrentUser(): Promise<User | null>`.
- Rate limiting on login endpoints.
- OpenAPI: document `/api/me` and any custom auth endpoints.

## 10. UI / UX

- **New pages:** `/login`, `/signup`, `/logout` (action), `/forgot-password`
  (stub acceptable this story if reset is deferred — note in Open Questions).
- **Modified:** `TopBar` shows real user avatar/name + logout; `AppLayout`
  enforces the auth gate.
- **Flows:** (1) signup → session → dashboard; (2) login → dashboard;
  (3) logout → login.
- **States:** loading (submitting), error (invalid credentials, generic message),
  empty (first run), offline.
- **Responsive:** auth forms centered, mobile-friendly.
- **Accessibility:** `<label>` per field, `aria-describedby` for errors, focus
  moves to first error on submit.
- **Copy & i18n:** all strings keyed.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Rolling custom auth introduces vulnerabilities | M | H | Use a vetted provider (Clerk/Auth0/Auth.js); no hand-rolled crypto |
| Session fixation / CSRF | L | H | Provider defaults + SameSite cookies + CSRF tokens; security test in F1.6 |
| Privileged data leaking into auth logs | L | H | Log allowlist; exclude request bodies; review log fields |
| Provider lock-in | M | M | Isolate behind `getCurrentUser()` seam so provider can be swapped |

## 15. Rollout Plan

- Feature flag: `auth_required` (default ON in all envs once merged).
- Sequencing: add provider + `users` table → wrap routes → add UI → enable gate.
- Pilot: internal accounts first.
- GA criteria: all ACs pass; security checks (F1.6) green.
- Rollback: disable `auth_required` flag to restore open access (dev only) while
  fixing; never disable in production with real data.

## 16. Test Plan

- **Unit** — session parsing, current-user helper, redirect logic.
- **Integration** — login/logout against the provider; `/api/me` 200/401 cases.
- **End-to-end** — Playwright: redirect when logged out (AC-1), login (AC-2),
  logout (AC-3).
- **Security** — authz matrix (public vs protected routes), rate-limit test,
  user-enumeration check, cookie-flag assertions.
- **Accessibility** — axe on auth pages; screen-reader pass on login form.
- **Performance / load** — login endpoint under concurrent load; auth-check latency.
- **Manual exploratory** — expired/tampered cookie, back-button after logout.
