## 1. Problem Statement

The dashboard is the attorney's home screen, but today it renders hardcoded rows
("Master Service Agreement", "94% Compliant") that mean nothing. Attorneys cannot
see their real contracts, statuses, or pending work, so the product has no daily
utility. This story replaces the mockup with a live, user-scoped dashboard backed
by the database.

## 2. Goals

- Show the signed-in user's real contracts with status, client, date, and
  compliance summary.
- Surface real "pending reviews" and high-level stats (active reviews, compliance
  rate, next renewal).
- Provide entry points to open a contract and to upload a new one.
- Keep the established DESIGN.md look while wiring everything to real data.

## 3. Non-Goals

- Upload/ingestion pipeline itself (C2.2) — the "Upload" button may link to it or
  be disabled until C2.2 ships.
- Contract detail/version pages (C2.3) and the editor (E3.x).
- The compliance computation (L4.x) — the dashboard *displays* stored scores; it
  does not compute them.

## 4. Personas & User Stories

- **As an associate attorney**, I want to see my active contracts and what needs
  review so I can prioritize my day.
- **As a partner / reviewer**, I want at-a-glance portfolio stats so I can gauge
  risk across matters.
- **As a legal-ops admin**, I want the dashboard scoped to permissions so users
  only see contracts they may access.

## 5. Functional Requirements

- **FR-1.** The dashboard MUST list contracts the current user is authorized to
  see, newest first, with name, client id, date, and compliance badge.
- **FR-2.** The dashboard MUST show summary stats (active reviews count,
  compliance rate, next renewal) computed from real data.
- **FR-3.** The dashboard MUST show a "pending reviews" indicator reflecting real
  counts.
- **FR-4.** Each contract row MUST link to its detail view (or be disabled with a
  tooltip until C2.3 ships).
- **FR-5.** The list MUST paginate or lazy-load beyond a threshold (e.g. 25 rows).
- **FR-6.** The dashboard MUST render distinct empty, loading, and error states.
- **FR-7.** The list SHOULD support basic sorting by date and status.

## 6. Non-Functional Requirements

- **Performance** — dashboard data query p95 < 300ms for ≤ 1k contracts; initial
  render (server) < 1s.
- **Security** — every query MUST filter by the authenticated user's access scope;
  no IDOR (cannot fetch another user's contract by id).
- **Privacy & Compliance** — only metadata on the dashboard; no privileged
  document body text rendered here; access is auditable (P5.3 hook).
- **Accessibility** — table is a semantic `<table>` with headers; badges have text
  equivalents (not color-only); WCAG 2.1 AA contrast.
- **Scalability** — server-side pagination; indexed queries.
- **Reliability** — partial failures (e.g. stats service) degrade gracefully
  without blanking the whole page.
- **Observability** — log dashboard load latency and result counts.
- **Maintainability** — data access via a typed repository/service layer.
- **Internationalization** — dates, numbers, and copy localized.
- **Backward compatibility** — replaces the static mockup; remove it after parity.

## 7. Acceptance Criteria

- **AC-1.** *Given* a user with contracts, *When* they open `/dashboard`, *Then*
  their real contracts appear newest-first with correct metadata.
- **AC-2.** *Given* a user with no contracts, *When* they open `/dashboard`,
  *Then* an empty state with an upload call-to-action shows (no fake rows).
- **AC-3.** *Given* User A, *When* they request User B's contract id directly,
  *Then* the API returns 403/404 and nothing leaks.
- **AC-4.** *Given* the data query fails, *When* the page loads, *Then* an error
  state renders and the failure is logged.
- **AC-5.** *Given* > 25 contracts, *When* the user scrolls/paginates, *Then*
  additional pages load correctly.

## 8. Data Model

- Uses `contracts` table (owned by F1.3, referenced here): `id` (uuid pk),
  `owner_id` (fk → users), `name`, `client_id`, `status`
  (enum: `draft|in_review|compliant|at_risk|pending`), `compliance_score` (int,
  nullable), `renewal_date` (date, nullable), `created_at`, `updated_at`.
- Indexes: `(owner_id, created_at desc)`, `(owner_id, status)`.
- If `contracts` does not yet exist, this story includes the migration
  `prisma/migrations/<timestamp>_add_contracts/migration.sql`; otherwise it
  depends on F1.3 for it.
- Backfill: seed sample contracts for demo/dev only (never in production).

## 9. API Surface

- `GET /api/contracts?cursor=&limit=&sort=` → paginated list scoped to the user:
  `{ items: Contract[], nextCursor?: string }`.
- `GET /api/contracts/stats` → `{ activeReviews: number, complianceRate: number,
  nextRenewal: string | null, pendingReviews: number }`.
- All endpoints require an authenticated session (F1.2) and enforce ownership.
- Rate limiting per user.
- OpenAPI: document both endpoints and the `Contract` schema.

```ts
type Contract = {
  id: string;
  name: string;
  clientId: string;
  status: "draft" | "in_review" | "compliant" | "at_risk" | "pending";
  complianceScore: number | null;
  renewalDate: string | null; // ISO date
  createdAt: string;          // ISO datetime
};
```

## 10. UI / UX

- **Modified page:** `/dashboard` (ported from `project_dashboard/code.html`).
- **Components:** `StatsCards`, `ContractsTable`, `ComplianceBadge`,
  `EmptyState`, `ErrorState`, `UploadButton`.
- **Flows:** (1) view contracts; (2) open a contract; (3) start an upload;
  (4) sort/paginate.
- **States:** loading skeletons for cards + table; empty state with CTA; error
  state with retry; offline notice.
- **Responsive:** table → stacked cards on small screens (DESIGN.md reflow).
- **Accessibility:** sortable headers expose `aria-sort`; badges include text;
  row links are keyboard-focusable.
- **Copy & i18n:** all labels and the welcome line keyed; pluralize counts.
- **Tokens:** reuse the existing badge/card/table styles from DESIGN.md.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| IDOR / cross-tenant data leak | M | H | Enforce `owner_id` filter in repository; authz test AC-3 |
| N+1 queries on stats | M | M | Aggregate in SQL; add indexes |
| Mockup parity drift | M | L | Visual diff against `project_dashboard/screen.png` |
| Empty/error states forgotten | M | M | ACs require them; covered in tests |

## 15. Rollout Plan

- Feature flag: `dashboard_live_data` (default OFF until parity verified, then ON).
- Sequencing: add/confirm schema → endpoints → wire UI behind flag → verify → flip.
- Pilot: internal/dogfood accounts with seeded data.
- GA criteria: all ACs pass; performance targets met.
- Rollback: turn off the flag to fall back to a static placeholder.

## 16. Test Plan

- **Unit** — stats calculations, badge/status mapping, pagination cursor logic.
- **Integration** — `GET /api/contracts` scoping + pagination; `stats` accuracy.
- **End-to-end** — Playwright: populated dashboard (AC-1), empty state (AC-2),
  pagination (AC-5).
- **Security** — IDOR test (AC-3); ensure list filters by owner; authz matrix.
- **Accessibility** — axe on the dashboard; keyboard nav of table + sort.
- **Performance / load** — query latency with 1k seeded contracts.
- **Manual exploratory** — mixed statuses, missing scores, null renewal dates.
